# 方法论:数据集构建与 schema 数据湖

## 1. 任务定义

本项目为 agent 驱动的 Text-to-SQL 场景准备了一份经过混淆的评测数据集。该数据集旨在削弱前沿语言模型因记住公开 BIRD schema 标识符而获得的优势,同时保留一个语义上仍然可用的 text-to-SQL 任务。设想中的场景是这样的:agent 先用已知的正确 SQL 配合 schema 元数据(仅包含列名和 dtype)构建语义记忆层,再针对正确的 schema 生成 SQL,回答新的自然语言问题。

评测会预先把正确的数据库交给模型。这里不评测路由;路由留给下游的评测框架处理。本项目的任务是产出一个让列名记忆不再那么直接有用的 schema 数据湖,而不是证明所有形式的基准记忆都已被消除。

---

## 2. 下游任务

本项目服务的下游任务,是一套横跨多数据库 schema 数据湖的 agent 驱动 Text-to-SQL 工作流。在构建记忆阶段,agent 会收到:

- 每个保留数据库的精简 schema 元数据(仅包含表名、列名和 dtype);
- 配有已知正确 SQL 的训练问题;
- 本仓库产出的经过混淆的 PostgreSQL schema。

在测试阶段,agent 会拿到一个新的自然语言问题,然后必须生成能在混淆后 schema 上正确执行的 SQL。下游评测框架可能还会额外评测 schema 路由、检索、记忆构建或 agent 规划。这些系统级的取舍不在本仓库范围之内;本仓库只负责准备并验证该任务所用的混淆数据产物。

---

## 3. schema 数据湖的构建

在做任何过滤之前,先把 BIRD 的 train 和 dev 划分合并起来。train 和 dev 之间没有重叠的数据库(共享的 db_id 为 0),因此合并后的语料共有 80 个不同的数据库、总计 10,962 个问题。

所有保留的数据库都会被载入两个干净的 **PostgreSQL 实例**(`pg_base` 和 `pg_rename`),如 [obfuscation.md §5](obfuscation-zh.md) 所述;另外两个带诱饵的实例(`pg_decoy`、`pg_rename_decoy`)是携带被破坏陷阱的克隆(共四个实例)。在每个实例中,每个 BIRD 数据库都映射到一个 PostgreSQL schema:

```text
world.country
movies_4.country
works_cycles.employee
...
```

这种完全限定的命名方式无需重命名或修改任何 schema,就解决了所有表名冲突。在 69 个保留的数据库中,有 45 个表名被不止一个数据库共用(例如 `country` 出现在 11 个数据库中)。PostgreSQL 的 schema 让这些表名在查询层面变得没有歧义。

BIRD 的 gold SQL 会从 SQLite 转译到 PostgreSQL 方言,并被改写为使用 schema 限定的表引用(`db_id.table_name`)。管线概览见 [obfuscation.md §5](obfuscation-zh.md),R0==R1 验证具体保证了什么(包括 VALUES 物化以及失败桶)见 [../reference/step5-transpilation.md](../reference/step5-transpilation-zh.md)。

## 4. 数据库纳入标准

数据湖中的每个数据库都会贡献两样东西:schema(在记忆构建阶段供 agent 使用),以及留出的测试问题(用于衡量评测准确率)。一个无法贡献有意义测试集的数据库,提供不了任何评测信号。

**最低阈值:每个数据库 60 个问题**,按 train+dev 合并后的计数来判定。

低于该阈值的数据库,问题数量不足以支撑一个能可靠覆盖各难度的分层留出集。该阈值在合并语料上判定,因为把 train 和 dev 合并,能让那些较小的、只出现在 train 中的数据库在被剔除前获得最大的通过机会。

全部 11 个被剔除的数据库在 train 划分中都不足 60 个问题,且在 dev 中没有对应项:

| 数据库 | 合并问题数 | 剔除原因 |
| --- | --- | --- |
| `craftbeer` | 6 | 实际上无法评测 |
| `citeseer` | 19 | 最小难度单元格 = 2 |
| `genes` | 23 | 没有 challenging 级问题 |
| `shooting` | 28 | 数量太少,无法形成有意义的留出集 |
| `trains` | 40 | 处于临界;无法按难度划分 |
| `music_tracker` | 45 | 最小单元格 = 3 |
| `movie` | 46 | 没有 challenging 级问题 |
| `coinmarketcap` | 48 | 最小单元格 = 3 |
| `mental_health_survey` | 50 | 处于临界 |
| `european_football_1` | 57 | 处于临界 |
| `human_resources` | 59 | 处于临界 |

这 11 个数据库同时被排除在 schema 数据湖和评测之外。剩下的 **69 个数据库**(10,541 个问题)构成了该数据集。

## 5. 训练 / 测试划分

原始的 BIRD train/dev 划分被弃用。在合并后的问题池上,对 69 个保留数据库逐个重新做一次随机划分。

### 划分策略

对每个数据库做一次 **80/20 的随机留出**,并用固定随机种子保证可复现:

- 每个数据库 80% 的问题 → train(用于 agent 记忆构建)
- 每个数据库 20% 的问题 → test(留出用于评测)

### 依据

- **先合并再划分** 避免浪费原始的 BIRD dev 问题。就本项目而言,原始的 train/dev 边界是任意的。
- **库内划分** 适合下游的记忆构建场景,因为 agent 本来就应该在见过的 schema 上构建记忆。这样评测的是在保留数据库内部留出问题上的表现,而不是对未见过数据库的泛化能力。
- **随机而非分层**,因为 BIRD 的 train 划分不带难度标签;它们在 JSON 中根本不存在。BIRD 的 dev 难度标签是人工标注的,与从 SQL 复杂度推导出的标签不可比。在不一致的标签上做分层只会带来虚假的精确感。随机划分则诚实,也可复现。

---

## 6. 评测指标

每个测试问题只以执行准确率来评测。正确的数据库会预先交给模型;路由准确率不在本项目范围之内。

| 指标 | 定义 |
| --- | --- |
| **执行准确率(EX)** | 生成的 SQL 在 schema 数据湖上运行时,是否返回正确的结果集? |

既报告整体结果,也按以下维度拆分:

- 混淆语言(英语对照组 / 法语 / 德语 / 西班牙语 / 拼音)
- 冲突状态(表名出现在 ≥2 个数据库中,还是唯一)

### 最终验证交付物
