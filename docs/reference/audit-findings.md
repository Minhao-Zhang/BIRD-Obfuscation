## Why this audit exists

The pgloader rewrite of step 4 (`pipeline/04_load_pg_base.py`, see
`AGENTS.md`) was originally justified by a claim that some SQLite columns are
declared `INTEGER`/`REAL` but actually hold non-numeric strings (e.g.
`app_store.Price` = `"$4.99"`), breaking the old hand-rolled `NUMERIC`
mapping in `pipeline/_pg_helpers.py`. This audit checks that claim against
every retained database, and separately checks for identifiers (table/column
names) that aren't plain lowercase `[a-z0-9_]` — a real risk for any pipeline
step that assumes PostgreSQL will preserve the original SQLite spelling.

| Check | Result |

| Mixed-type columns (declared numeric, holds non-numeric strings) | **0** across all 69 DBs |
| Risky identifiers (non-lowercase / space / punctuation) | **2,351** across **48 of 69** DBs |

**Known gap in this "0 mixed-type columns" result (found 2026-07-01, during the
actual step-4 production run):** `is_numeric_value()` above only checks
whether a value is parseable as *some* number (`float(str(v))` succeeding) —
it does not check whether the value's numeric *shape* matches the column's
declared type. A column declared `INTEGER` that actually holds a float like
`182.88` (`european_football_2.Player.height`, apparently centimeters
converted from inches) passes this check cleanly, so this audit's "0" is not
a clean bill of health for that failure mode — it only ruled out non-numeric
*strings* in numeric columns, not floats in integer columns. This caused a
real step-4 COPY failure not caught in advance by this audit; see
`../methodology/obfuscation.md` §4 and `04_load_pg_base.py`'s `EXTRA_CASTS["european_football_2"]`
for the fix. If revisiting this script, `is_numeric_value` should also flag
`isinstance(v, float) and not v.is_integer()` for `INTEGER`-declared columns.

**The original type-mismatch justification does not hold up.** `app_store`'s
`playstore.Price` column is itself declared `TEXT` in SQLite, not
`INTEGER`/`REAL` — the `"$4.99"` values are exactly what a `TEXT` column
should hold. No database in the retained set has a column declared numeric
that actually contains non-numeric string data. Whatever caused the original
6-DB COPY failures under the old hand-rolled loader
(`ice_hockey_draft`, `mondial_geo`, `professional_basketball`, `talkingdata`,
`works_cycles`, `european_football_2` — per the now-deleted `CHECKPOINT.md`),
it was very likely **unquoted identifiers breaking DDL/COPY statements**, not
numeric coercion: those 6 DBs are exactly among the highest-risky-identifier
databases in the table below.

The identifier risk is real and large. Patterns found beyond simple
uppercase/lowercase:

- **Embedded spaces**: `app_store."Content Rating"`, `thrombosis_prediction."Examination Date"`, `synthea."POPULATION TYPE"`, `superstore."Customer ID"`
- **Punctuation/symbols inside identifiers**: `talkingdata."F23-"` / `"F43+"`, `hockey."R/P"` / `"+/-"`, `disney."Studio Entertainment[NI 1]"`, `retail_complains."Consumer consent provided?"`
- **Hyphenated table names** — not valid unquoted SQL identifiers at all: `legislator` (`current-terms`, `historical-terms`, `social-media`), `disney` (`voice-actors`)
- **Whole-schema PascalCase/camelCase**: `works_cycles` (65/65 tables affected), `hockey`, `mondial_geo`, `formula_1`

5. **Still the priority list to watch** once step 4 is actually run:

   `european_football_2`, `professional_basketball`, `synthea`,
   `thrombosis_prediction`, and `legislator` concentrate the identifier risk

   `legislator` and `disney` are additionally the only two DBs with
   hyphenated table names (`current-terms`, `historical-terms`,

## Open question this audit does not answer (resolved 2026-07-01)

Whether pgloader's default `downcase identifiers` behavior (documented for
SQLite sources, with no documented override unlike MySQL/PostgreSQL sources)
actually applies given the current `WITH create tables, create indexes,
reset sequences` clause in `pipeline/04_load_pg_base.py`. This audit only
scans the SQLite source files — it does not load anything into PostgreSQL.

**Resolved by reading pgloader's source directly** (`src/params.lisp`,
`src/utils/quoting.lisp`, `src/parsers/command-options.lisp`): the answer is
yes, `downcase identifiers` is the default and applies to SQLite sources —
but only for identifiers matching `^[A-Za-z_][A-Za-z0-9_$]*$`. Anything with
spaces/punctuation (already the focus of this audit) was always forced into
the quoted, case-preserved branch regardless of the WITH clause. The actual
gap was plain PascalCase/camelCase names with no punctuation — e.g. every
table in `works_cycles` — which passed that regex and were being silently
downcased. `pipeline/04_load_pg_base.py` now passes `quote identifiers`
explicitly in the WITH clause (confirmed valid grammar for SQLite sources,
not MySQL-exclusive as this audit originally speculated), and verifies the
result empirically after every load by diffing SQLite `PRAGMA table_info`
and `PRAGMA foreign_key_list` against `pg_base`'s `information_schema`
— so a future pgloader version change would be caught immediately rather
than surfacing later as an R0==R1 execution failure. See `../methodology/obfuscation.md`
§4 "Identifier quoting invariant" for the full writeup, including why
foreign keys needed a separate check from table/column identifiers.

**Unrelated follow-up (same day):** `reset sequences` was separately removed
from the WITH clause (not because of casing — because of a distinct pgloader
v3 bug where `quote identifiers` + `reset sequences` together produce a hard
`42703` error on mixed-case serial/PK columns; see `../methodology/obfuscation.md` §5 step 1
and the code comment on `pgloader_command_script()` in
`pipeline/04_load_pg_base.py`). The WITH clause quoted above
(`create tables, create indexes, reset sequences`) reflects the clause as it
stood when this audit was written, not the current one.
