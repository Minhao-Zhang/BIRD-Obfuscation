# 损坏诱饵设计:风险登记册与决策

**状态:** 已实现(两个阶段均已注入 `pg_decoy`/`pg_rename_decoy`)。
本文记录了设计方案、已知的坑,以及(§5)实际落地的参数。风险登记册(§2)
和决策(§3)作为 §5 的依据保留下来。

**(有条件地)取代:** [extension-implementation-plan.md §0](extension-implementation-plan-zh.md)
中 `Populate decoy tables with rows? → No` 这一默认设定。该默认设定对于**单轮
(single-shot)**的 text-to-SQL 评测是正确的(模型从不探测,因此空诱饵既不可见也不
产生任何代价)。但换成**交互式 / 执行并观察的 SQL 智能体**,它就成了**缺陷**:
诱饵表上的 `COUNT(*)=0`,或某个 100%-NULL 的诱饵列,会立刻暴露诱饵,使其贡献的
干扰压力为零。本文档假定交互式范式才是目标(已于 2026-07-04 确认)。**范围:** 本仓库
*唯一*的职责是产出 Postgres 数据库(数据集)加上一份适用于该任务的**陷阱清单(trap
manifest)**;交互式评测框架位于一个**独立的仓库**中。

---

## 1. 概念:「copy → rename → corrupt」

不使用空表 / NULL 列,而是从一个**真实的**列(或表)构建每个诱饵:

1. **复制(Copy)**一个真实列 `C` → *免费*获得逼真的类型、空值比例、基数、取值
   分布(比从零合成要容易得多,也更有说服力)。
2. **重命名(Rename)**这份副本,使用一个看似合理的*同义词*(`price → list_price/unit_cost`,
   而不是 `price_copy`)。
3. **损坏(Corrupt)**这份副本,使其存在细微的错误,从而把诱饵从一个惰性的*干扰项*
   变成真正的*陷阱*(选中它的智能体会得到错误答案)。

**损坏算子**,位于隐蔽 ↔ 可靠出错这一光谱上:

| 算子 | 单列画像能否检测到? | 陷阱作用于 |
| --- | --- | --- |
| **置换(Permutation)**(把已有取值重新分配给其他行) | 否:min/max/mean/histogram/distinct 全都相同 | 仅逐行的查找 / 连接(对聚合不可见) |
| **稀疏扰动(Sparse perturb)**(损坏 p≈10-30% 的行) | 勉强能:聚合值会略微移动 | 被改动的那 p% 的行 |
| **全局扰动(Global perturb)**(`C±ε`、`C·(1±ε)`,所有行) | 能,如果智能体对取值范围有预期 | 几乎总是 |
| 域内类别重映射 / 日期偏移 / 舍入 / 单位偏移 | 视情况而定 | 视情况而定 |

适用于两种粒度:**邪恶双胞胎列(evil-twin column)**(添加到真实表上)和**损坏克隆表
(corrupted clone table)**(克隆一张真实表,重命名该表及其各列,并损坏其中一部分列)。

## 2. 风险登记册

### A. 校验不变量风险(不得破坏 R1==R2 或 gold)
- **A1:`SELECT *` 泄漏。** 损坏 / 已填充的诱饵列之所以安全,*仅仅*是因为 gold
  从不引用它们,而唯一的泄漏路径(`SELECT *`)已被 `gold_star_expanded` 剥离。如果
  该快照过时或不完整(某个 `SELECT *` 的 gold 未被捕获),损坏列就会泄漏进展开后的
  gold → R1≠R2 → gold 出错。*缓解措施:* 断言每一个 `SELECT *` 的 gold 都有对应的
  展开;重新运行完整的门禁。
- **A2:比较器依赖。** 重新校验依赖于已修复的 `normalise_result`(发现项 A,目前
  **尚未提交**)。请先提交它,否则虚假的顺序敏感不匹配会掩盖真实的不匹配(或冒充成
  真实的不匹配)。
- **A3:重新注入的顺序。** `gold_star_expanded` 在注入*之前*读取真实列。任何对诱饵的
  重新生成都必须在重新注入**之前**重新快照,否则会把诱饵列当作真实列捕获下来。

### B. 可检测性破绽(会破坏诱饵的目的)
- **B1:域内合理性。** 超出取值域的损坏值(负的 `age`、未来的 `birth_date`、越界的
  编码)会被画像识破。应逐列进行钳制 / 约束。
- **B2:聚合破绽。** 全局扰动会改变 min/max/mean/sum。为求隐蔽,优先采用稀疏 / 置换。
- **B3:键 / 唯一性。** 对唯一列 / ID 列加噪会产生碰撞或悬空引用 → 看起来是假的。对
  类键列应使用**置换**(保持唯一性,每个值都仍是真实值;同时也是极好的*连接*陷阱)。
- **B4:格式 / 模式。** 破坏了某列隐含格式的损坏字符串(邮箱、电话号码、邮政 / ISO
  编码)一眼就能看出。损坏必须是格式感知的。
- **B5:跨列 / 跨行不变量(最重要的一条)。** 真实列往往是相关的:`total = qty × price`、
  `start_date ≤ end_date`、`city/state/zip` 的一致性、派生 / 反规范化列。损坏一个相关
  列集合中的**某一**列会破坏该不变量 → 会做一致性检查的智能体就能发现造假。对相关列
  而言,孤立的逐列损坏是不安全的。*缓解措施:* 检测并跳过相关列,或者把整个一致的
  集合一起损坏。
- **B6:标定。** 太弱 → `C'≈C`(只是一份副本:可疑,且构不成陷阱)。太强 → 明显是假
  的。损坏率需要调优。
- **B7:空值比例异常。** 增删 NULL 会把空值比例改变到一个不合理的水平。
- **B8:孤立的诱饵表。** 未被引用的克隆表在模式探索中可能显得可疑。「处处无外键」的
  设计部分地掩盖了这一点,但孤岛仍可能突兀显眼;可以考虑用相关表里的真实 ID 来填充
  一个看似合理的键列。
- **B9:外键 / 连接键完整性(跨表)。** *真实的* gold 连接**不**受威胁。陷阱是附加性
  的,从不触碰真实键(R1==R2 已确认每个连接在诱饵实例上都被保留)。风险在于**陷阱
  自身的**键:对连接键 / 类 FK 列做朴素的值加噪会产生**悬空引用**(找不到匹配父行的
  键),而参照完整性检查(`fk NOT IN (SELECT pk)` → 孤儿行)会借此揭穿诱饵;此外,对
  子表 / 父表不一致地下陷阱(其中一张 ≤N,另一张作为巨表被跳过)会让孤儿行更加显眼。
  这个数据库**没有声明任何 FK**(加载时用了 `no foreign keys`),因此连接键必须靠
  *推断*(名称模式 `*_id`、类型 + 值子集匹配),并不完美,所以我们可能漏掉某个键
  并把它损坏了。*缓解措施:* 把推断出的连接键 / FK 列当作相关列处理(决策 4):要么
  跳过它们,要么**仅通过置换**来损坏(每个值都仍是一个真实、已存在的键 ⇒ 参照完整性
  得以保持,同时仍是一个隐蔽的连接陷阱;见 B3)。克隆表的键列同样必须保留有效的父键。

### C. 陷阱有效性风险
- **C1:隐蔽性与可靠出错之间的张力。** 稀疏 / 置换损坏可能**不会**与某个给定问题触及
  的行重叠 → 陷阱悄无声息地不触发,诱饵对该问题毫无作用。必须在两种理念之间抉择:
  *概率性 + 测量一个陷阱触发率*,还是*稠密 / 定向(可靠,但会把诱饵与特定问题耦合起来)*。
- **C2:算子 ↔ 查询不匹配。** 置换对聚合不可见(对置换列做 `SUM`,结果不变);它只能
  陷住逐行的操作。要让算子与你打算陷住的查询类型相匹配。
- **C3:信号大小。** 太稀疏 → EX 几乎不动,各实验臂无法区分;太稠密 → 不真实。要标定
  到一个可测量但又合理的效果。

### D. 可复现性 / 确定性
- **D1:稳定的行标识。** 基于哈希的确定性损坏需要一个稳定的主键。无主键的表 →
  `ctid` 在重新加载 / 重建时**不**稳定 → 不可复现。退而求其次,在一个固定的*全序*上
  用确定性的 `row_number()`(存在并列时会产生歧义;有些表没有天然的稳定顺序)。
- **D2:按语义而非名称设定种子。** 损坏的种子应基于 `(db, source_column, operator)`,
  **而不是**诱饵在各变体下(重命名后)的名称。否则 `pg_decoy`(英文)和
  `pg_rename_decoy`(目标语言)会以与重命名无关的方式产生分歧,从而**混淆消融实验**。
- **D3:不使用 PG 的 `random()`。** 它不可复现,而且重新克隆会重置诱饵数据卷。改用
  基于哈希的伪随机(例如 `('x'||md5(pk::text))::bit(32)::int`),这样重建就能复现出
  完全相同的数据集。

### E. 成本 / 运维
- **E1:存储。** 填充数据会逆转当前「免费」的特性。数百万行大表(例如 `ratings`)上的
  邪恶双胞胎列每个都要花费一整列的存储;克隆表则要花费整张表的数据量。这可能会增加
  好几 GB。要做预算并进行测量。
- **E2:写入成本与膨胀。** 在一张巨表上用 `UPDATE` 填充一列会产生大量 WAL,而且在
  `autovacuum=off` 的情况下死元组永远不会被回收 → 数据卷的增长*超过*数据本身。可能
  需要执行 `VACUUM FULL`(会重写整张表,需要空闲空间),或者接受一个膨胀预算。
- **E3:幂等性 / 重复运行。** `inject_variant` 目前会*跳过*已存在的表 / 列,因此重复
  运行**不会**更新损坏。需要一条 `--regenerate` / 强制路径来删除并重建诱饵对象。
- **E4:完整重新校验。** 每次诱饵数据变更都需要重新运行 R1==R2 门禁(每个变体约
  45 分钟,为避免 OOM,一次只跑一个变体)。

### F. 工作次序 / 范围
- **F1:先确定范式,再动手构建。** 损坏诱饵只有在交互式的执行→观察→改进框架下才有
  意义,而这个框架**目前还不存在**(`eval_ablation.py` 是单轮的)。在交互式框架就绪
  之前就去构建损坏诱饵可能为时过早。要确认这个框架确实会到来。
- **F2:陷阱触发率指标需要一份新的产物。** 要衡量智能体是否被陷住,需要*与诱饵一致
  的答案*(用 `C'` 而非 `C` 计算出的 gold)。如果我们想要一个陷阱触发率指标,就得
  定义并计算它。
- **F3:与污染问题正交。** 模型可能在预训练中就已经知道了真实的 BIRD 取值;损坏一份
  副本并不能解决记忆问题。这与本工作正交,但需记录在案。

### G. 多语言 / 变体
- **G1:数据语言。** 诱饵列的*取值*是从真实列复制而来的,而这些真实列的**数据**使用
  的是该数据库的语言(对于非英文数据库)。重命名只触及标识符,不触及数据,因此字符串
  损坏(错别字、类别重映射)必须尊重数据的语言 / 字符集。

## 3. 决策(于 2026-07-04 敲定)
1. **范式。交互式 SQL 智能体:是**,但本仓库只产出 Postgres 数据库加陷阱清单;评测
   框架是一个**独立的仓库**(见上文范围)。⇒ **F1** 已解决(此处不涉及框架工作);
   **F2** 在下游计算,但本仓库**必须产出该清单**。
2. **下陷阱的理念。两者都做:混合多种陷阱类型。** 在各列间支持并混合*隐蔽型*
   (置换 / 稀疏扰动)与*可靠型*(更稠密 / 定向)算子,而不是单一风格(C1-C3)。
3. **范围:两者兼有** —— 既有邪恶双胞胎列*也有*损坏克隆表(二者混合)(E1、B8)。
4. **相关列:允许作为陷阱来源(已于 2026-07-04 反转)。** 最初的「跳过」来自 B5
   (损坏 `total=qty*price` 集合中的某一列会破坏不变量)。但最终设计是严格**附加性**
   的(我们从不修改真实列,只是以新名字添加一份损坏的*副本*),因此真实的跨列不变量
   始终得以保留,**B5 无法触发**。相关列(价格、数量、日期)是价值最高的陷阱目标,
   所以它们确实是合格的来源。**连接键 / FK 列仍然只做置换(B9 依然成立)。**
5. **损坏率:每个陷阱列损坏 10% 的行**(稀疏)(B6、C3)。
   ⚠️ *确认解读:* 「10 percent」= 每个陷阱列损坏 10% 的**行**(而不是把 10% 的列
   变成陷阱)。按「行」这一解读推进。
6. **存储:跳过大表。** 只向**≤ N 行**的表注入陷阱;跳过任何更大的表,以此约束
   UPDATE / 存储 / 膨胀成本(E1-E2)。
   调研(2026-07-04;569 张真实表,合计约 362M 行):494 张表(87%)在 ≤100k 行,却
   只占全部行的 **1.2%**;532 张(93.5%)在 ≤500k;37 张 >500k 的表占了 **~99%** 的
   数据(`language_corpus.pages_mots` 129M、`bike_share_1.status` 72M、`talkingdata…`
   32M、`movie_platform.ratings` 15.5M、……)。**N = 500,000(于 2026-07-04 锁定)** →
   532/569(93.5%)覆盖率,只跳过那 37 张巨表。克隆表来源(会复制所有列)很可能需要
   一个**更低**的上限或一个计数限制。

## 4. 硬性不变量与前提条件
- **原始数据永不被改动。诱饵严格是附加性的。** 陷阱是*添加*出来的副本(新列 / 新表);
  每一个真实列 / 表都与干净实例保持逐字节相同。正是这一点既让正确答案在诱饵数据库中
  仍然可达,**又**保持了 R1==R2。*(设计要求:在下了所有这些陷阱之后,原始数据必须
  仍然可用。)*
- **产出一份陷阱清单**(每个陷阱:`db`、真实来源表 / 列、诱饵名称、算子、参数、种子)。
  它是那个独立评测仓库所需的基准事实,用于(a)知道哪个是陷阱、哪个是原始,以及
  (b)为陷阱触发率指标计算与诱饵一致的答案(F2)。
- 在重新校验之前提交 `normalise_result` 的修复(A2)。
- 永远不要改动 `pg_base`/`pg_rename` 或 `*_final.jsonl`(扩展计划的黄金法则)。
- 在任何重新注入之前,重新快照 `gold_star_expanded`(A3)。

## 5. 实现(实际落地)

驱动脚本:[`pipeline/10_inject_traps.py`](../../pipeline/10_inject_traps.py);算子
[`pipeline/_corruption.py`](../../pipeline/_corruption.py);规划器 / 应用器
[`pipeline/_traps.py`](../../pipeline/_traps.py)。两种附加性的陷阱粒度:

**阶段 1:邪恶双胞胎列**(`--phase rowcounts,plan,name,inject`)。真实表上的一个新列,
以 LLM 给出的同义词命名,持有某个真实*来源*列的损坏副本。表容量上限**≤ 500 000 行**
(决策 6);每张表最多 3 个陷阱,优先选择查询中用到的列(连接键 + 相关列)。清单:
`artifacts/trap_manifest.json`(1486 个陷阱)。

**阶段 2:损坏克隆表**(`--phase plan-tables,name-tables,inject-tables`)。把一整张真实表
克隆出来(`CREATE TABLE … AS SELECT … FROM real`),表名和列名都用 LLM 给出的同义词,
然后损坏其中一部分列。表容量上限**≤ 50 000 行**(`CLONE_ROW_CAP`);每个数据库克隆
**符合条件表的 30%,2-8 张**(`CLONE_FRAC`、`CLONE_MIN/MAX`);在一张克隆表内,损坏
每个连接键 + 约一半的其余列,其余保持精确以形成逼真的混合。清单:
`artifacts/trap_table_manifest.json`(162 张克隆表 / 66 个数据库 / 1228 列,其中 724 列
被损坏)。**由构造保证 R1==R2 安全**:gold 从不引用诱饵表,因此克隆损坏不可能改变
gold 的结果。

**算子**(`choose_operator`,按决策 2 做确定性混合):连接键 → 仅 `permute`(B9/B3);
数值 → `permute`/`sparse_perturb`;文本 → `permute`/`sparse_cat_remap`;时间 →
`sparse_date_offset`;其余 → `sparse_null`。所有损坏都是逐行键 + 一个**与变体无关**
的盐(salt)(D2/D3)的纯哈希函数,当存在单列 PK/唯一键时以其为键,否则用 `ctid`
(克隆表始终使用 `ctid`)。对整数做 `sparse_perturb` 时,加法在 `numeric` 中计算并
**钳制到目标整数类型的范围内**,这样大的 `bigint`(或赋值时的 `int4`/`int2`)就不会
溢出。`pct=0.10`(决策 5),`rel=0.15`。

**验证。** 已证明真实数据在两个诱饵数据卷上与干净实例逐字节相同(对全部 532 张下了
陷阱的表做与顺序无关的逐列指纹)。克隆表:行数与来源一致,精确复制列完全相同,置换
列作为多重集相同但 100% 行都被移动过,稀疏列已被改动。阶段 1 的 R1==R2 留下 153 个
良性的顺序敏感 qid + 21 个既有的执行失败 qid(`--regenerate` 会重新运行;已在
`artifacts/order_sensitive_qids.json` 中标记)。

## 6. 命名、已知局限与清单结构

**命名**(`--phase name` / `name-tables`,模型 `gpt-5.4-mini`,通过 `.env` 中的
`OPENAI_API_KEY`)。每个 (db, variant) 调用一次 LLM,索取一个**有序的 JSON 数组**
同义词;结果按位置分配,并做去碰撞处理以避开真实标识符和已用过的诱饵名称。base 变体
= 英文;rename 变体 = 该数据库的目标语言(多语言,含拼音)。在大数据库上,模型可能
返回一个**偏短的数组**,于是尾部会回退到 `<name>_alt` / `<table>_archive` 占位符:
一个统一的、明显是合成的后缀,执行并观察型智能体可以借此揭穿诱饵。缓解措施:只对
受影响的列重新生成,并**每次调用分块 ≤ 20 个**(`scratchpad` 式的一次性处理,模型
`gpt-5.4-mini`),使数组不会被截断。已交付的清单中有 **0** 个 `_alt`/`_archive`/`_copy`
名称(已对照运行中的诱饵实例核实)。

**已知局限(未修复):**
- *大小写风格的破绽。* 在使用 PascalCase 的数据库上(例如 `works_cycles`,
  AdventureWorks 风格),诱饵标识符是 snake_case,和 PascalCase 的真实列并排在一起,
  构成风格上的破绽。整个命名步骤都是统一如此;要修复需要具备大小写感知的名称生成。
- *陷阱触发覆盖(C1)。* 稀疏 / 置换陷阱只影响某个问题实际触及的行 / 操作;某个给定
  问题可能命中不了任何陷阱。这是有意为之的设计(在下游测量一个陷阱触发率),而非缺陷。

**清单结构**(交接给独立评测仓库的基准事实)。两份都位于 `artifacts/`。**盐**与变体
无关(采用英文来源身份),因此 `pg_decoy` 和 `pg_rename_decoy` 会以相同方式损坏相同
的行(D2);`names.base` 是英文诱饵标识符,`names.rename` 则是目标语言的标识符。

- `trap_manifest.json`:邪恶双胞胎**列**;每个陷阱一条记录:
  `{db, table, source_column, source_type, operator, is_key, in_correlated_group,
  salt, names:{base, rename}}`。添加到 `<db>.<table>` 上的诱饵列是 `names.<variant>`;
  它是 `source_column` 的一份损坏副本。
- `trap_table_manifest.json`:损坏克隆**表**;每张克隆表一条记录:
  `{db, source_table, columns:[{source_column, source_type, operator, is_key}],
  names:{base:{table, columns:[…]}, rename:{table, columns:[…]}}}`。列 `columns[i]`
  映射到 `names.<variant>.columns[i]`;`operator:null` = 精确复制(未损坏)。
- `order_sensitive_qids.json`:要从严格 EX 评分中排除的 qid(153 个顺序敏感 + 21 个
  既有的执行失败)。

要计算一个**与诱饵一致的答案**(陷阱触发率指标,F2),评测仓库会改写 gold SQL,让它
读取 `names.<variant>` 来替代 `source_column`(或用克隆表替代 `source_table`),然后
在诱饵实例上运行它。
