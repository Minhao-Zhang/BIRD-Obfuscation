# 已知局限与适用范围说明

本数据集既作为 text-to-SQL 的**资源**,也作为污染 / 鲁棒性的**测量工具**,以下如实列出它的各项局限。其中若干是外部评审时暴露出来的;凡是已在代码中得到缓解的局限,均加以注明。这些局限都不会推翻基于 delta 的消融结论,但它们界定了这些数字应当如何解读和报告。

## 1. 英文数据库不做标识符重命名(它们是对照组)

69 个数据库中有 14 个被指定为 `english`「语言」,因此它们的重命名映射完全是空操作:对这些数据库而言,`pg_rename` 与 `pg_base` 完全相同,且 `sql_rename == sql_base`。按问题数加权,这占语料的约 23%(≈2,326 / 10,164 行)。这是**有意为之**的。英文是重命名维度上的**噪声底线对照**(一个未经重命名的分支应当显示出 ≈0 的重命名 delta)。

对报告的影响:

- **应按语言分别报告重命名效应,以英文作为零效应基准,绝不要汇总成单一的 delta。** 汇总后的「重命名效应」在结构上被约 23% 必然为零的行所稀释,会低估真实的分语言效应。
- 这些数据库仍然会接受**完整的诱饵陷阱和复述**处理,因此整体上它们**并非**未经混淆。只有*重命名这一维度*对它们是空操作。「23% 与公开 BIRD 逐字节相同」只对 `pg_rename` 成立,对 `pg_decoy` / `pg_rename_decoy` 或复述分支都不成立。
- 如果某个使用场景需要*每一行*都能探测标识符记忆,可以对英文数据库做同义词重命名(英文→英文同义词)。目前没有这样做,因为那会去掉干净的噪声底线对照,这是一个有意的取舍。

## 2. 结果比较对类型宽松(BIRD 风格)

默认的比较器(`normalise_result`)会把每个单元格强制转换为浮点数,否则转为去除首尾空白并转小写的字符串。因此 `1 == "1" == True == Decimal("1.00")`,且字符串大小写被忽略。这符合 BIRD 的精神,并且**对每个分支都对称地应用**,因此它**在污染 / 消融的 delta 中相互抵消**(即核心信号)。不过,它可能**高估绝对 EX**(模型的 `"1"` 会匹配标准答案的 `1`)。

缓解措施(已实现):评测现在会**在宽松 EX 旁一并报告严格 EX**(`normalise_result_strict`:不做跨类型合并,保留数值相等性,大小写敏感,仍去除 CHAR 的填充空格),作为一个保守下限。在陈述绝对准确率时请引用严格那一列。NaN/inf 现在在 `normalise_result` 中也被规范化,因此它与第 5 步的比较器保持一致,返回的 NaN 不再会造成虚假的 `R1!=R2`。

## 3. 验证纳入设有执行预算

一个标准答案要被纳入交付物,需要在 `statement_timeout`(现为 **300 s**)和 **200,000 行**的抓取上限下,针对一个刻意不建索引的 schema 执行通过。行数上限是确定性的;而超时在边界处不是。一个恰好在超时线之下与之上完成的标准答案,在不同运行之间可能会翻转。实践中这只涉及少数几个**退化**的标准答案(缺失连接导致的笛卡尔积,例如一个 19.4M 行的标准答案),它们在 `artifacts/order_sensitive_qids.json` 中被记为 `exec_failed`。300 s 的余量使得对合法查询而言纳入实际上是确定的;在 BIRD 中,一个真实的 text-to-SQL 答案返回 >200k 行的情况基本不存在。

## 4. 重命名映射的冲突会被检测到,但需手动解决

第 3 步会对取值冲突发出警告;第 6 步逐个重命名数据库,遇到任何错误就回滚该数据库(使其保持未重命名)。跨表的列名冲突在 PostgreSQL 中是无害的(列的作用域限于所在表),并且是随附映射中唯一存在的一类冲突。而**同一作用域内**的冲突(表内两个标识符 → 同一名称,或两个表 → 同一名称)会导致 `ALTER` 失败。

缓解措施(已实现):第 6 步现在会**以非零状态退出并列出所有失败的数据库**,因此不可能出现某个数据库被悄悄丢弃的情况。这个信号很明显,修复办法是在映射中解决冲突后重新运行。自动的冲突解决(加后缀)没有实现。

## 5. 由 `VALUES` 物化的标准答案

约 11.5% 的标准答案含有 `VALUES` 子句,但其中只有**约 0.5%** 是真正不涉及表的(常量被内联写死,此时 `R0==R1` / `R1==R2` 是平凡地自我满足的);其余的都合理地使用 `VALUES`(作为 in-list / 派生表,与真实的表一起使用)。因此把它读成「约 12% 是循环验证」会是夸大其词。真正为常量的集合约为 0.5%。这些常量标准答案中有少数是非常大的字面量堆砌(多达约 4.4M 个字符):它们是忠实的转写,但并非自然的 SQL,作为供记忆学习的下游智能体使用的「已知为真的 SQL」也很差。请把它们当作边缘案例。

## 6. 一些次要说明

- 重命名短路会跳过任何不含带引号的重命名映射键的 SQL。它之所以安全,**仅仅**是因为每个标识符在上游都已加引号(这是一个有文档记录的不变式);一旦有未加引号的标识符到达它,它会以「放行」方式失败(悄无声息地成为空操作)。
- 证据提示的重命名是对自然语言文本(而非标准答案 SQL)做顺序的正则替换:风险很低;理论上,一个靠后的短键可能会匹配到已插入的译文内部。
- 即便是非英文数据库,也会通过恒等映射保留约 3-15% 的原始标识符(数字、本身已是母语的词、无法翻译的 token)。

## 关于完整性保证的精确说明

- **陷阱的可加性是由构造方式保证的**,而非靠运行时指纹:真实列只会作为损坏的*来源*被读取;每次损坏都写入*新的*诱饵列;克隆表都是全新的 `CREATE TABLE AS`。已提交的把关手段是 `R1==R2` 的重跑。(任何非正式提到的「指纹」比较都是临时性的验证,并非随附流水线的一部分。)
- **「真实数据逐字节相同」指的是真实列的*取值*未被修改**(逐列的多重集相同)。物理行的*顺序*和表的*列数*在诱饵实例上**并不**相同:第一阶段的 `ADD COLUMN` + `UPDATE` 会重写堆(重排物理行,这正是对于 `LIMIT`-无全序 / 浮点聚合的标准答案有 153 个顺序敏感排除项的原因),而新增的诱饵列会加宽 `SELECT *`。
